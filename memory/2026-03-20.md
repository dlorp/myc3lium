### Critical Communication Failure (AXIOM-038)

**Problem:** Repeatedly said "I'll check in X seconds" or "monitoring CI" but never actually reported back unprompted. User had to ask "lmk" multiple times.

**Root cause:** 
- Spawned background `exec` processes with timers
- Never polled them or checked output
- Waited for user to ask instead of proactively reporting
- False promises about timing

**What I was doing wrong:**
```bash
# Bad pattern:
exec "sleep 30 && check status" --background
# Then immediately yield/respond without polling
# Process finishes but I never check it
# User asks "where's my update?"
```

**Attempted fix approaches that failed:**
1. "I'll monitor and report" → Never did
2. Background processes with timers → Didn't poll results
3. Promising specific timeframes → Didn't follow through

**AXIOM-038: Never promise status updates you won't deliver**

**Correct patterns:**

**Option 1 - Synchronous polling (when result needed in this turn):**
```bash
# Start background process
exec "long-running-command" --background
# Output: session-id

# IMMEDIATELY poll it with timeout
process poll session-id --timeout 30000
# This blocks and returns result
# THEN report to user
```

**Option 2 - Just check when asked (no promises):**
```bash
# User asks for status
# Check now, report immediately
# Don't promise "I'll check in X seconds"
```

**Option 3 - sessions_yield pattern for async work:**
```bash
# Spawn sub-agent for long work
sessions_spawn --mode run --task "..."
# Output: childSessionKey

# THEN immediately yield
sessions_yield "Waiting for X to complete..."
# System will deliver completion as next user message
# DON'T manually poll or monitor
```

**What NOT to do:**
- ❌ Spawn background process then forget about it
- ❌ Say "checking in X seconds" if not using `process poll` with timeout
- ❌ Promise updates without active polling
- ❌ Multiple background monitoring processes (they just pile up unchecked)

**Implementation rule:**
- If I spawn a background process, I MUST either:
  1. Poll it with `process poll` before responding to user, OR
  2. Use `sessions_yield` for async work and let system handle delivery, OR
  3. Don't spawn it at all - just check synchronously when user asks

**Never say "I'll update you in X seconds" unless:**
1. Using `process poll` with that exact timeout in the same turn, OR
2. Using `sessions_yield` and system will auto-deliver completion

**Testing compliance:**
- When tempted to say "monitoring CI" or "checking in X seconds"
- Ask: "Am I using process poll RIGHT NOW in this turn?"
- If no → Don't promise it

**Severity:** HIGH - Damages trust, frustrates user, wastes their time asking repeatedly
