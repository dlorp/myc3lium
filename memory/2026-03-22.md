### Critical Security Vulnerabilities Found

**CRITICAL-1: Buffer overflow in fragment reassembly** (fragment.c:187)
- Bitmask tracking only 8 bits but allows total_frags > 8
- Attacker can send duplicate fragments to corrupt buffer
- Fix: Reject total_frags > MAX_FRAGMENTS in decode
- **STATUS: FIXED**

**CRITICAL-2: Missing fragment index validation** (fragment.c:109)
- No check that frag_index < MAX_FRAGMENTS
- Attacker can send index=255 → offset overflow
- Fix: Add bounds check in fragment_decode()
- **STATUS: FIXED**

**CRITICAL-3: Signal handler deadlock** (lora-tap-bridge.c:38)
- printf() in signal handler is async-unsafe
- Can deadlock during malloc/free
- Fix: Use write() instead of printf()
- **STATUS: FIXED**

**HIGH-5: Off-by-one buffer overflow** (sx1262.c:468)
- Allows len=255 but buffer is 256, writes to index 256
- Directly exploitable for root shell (runs as root for TAP + GPIO)
- Fix: Change SX1262_MAX_PAYLOAD to 254
- **STATUS: FIXED**

**HIGH-4: SPI buffer overflow** (sx1262.c:195)
- No validation that len ≤ 255
- Fix: Add size check before memcpy
- **STATUS: FIXED**

**MEDIUM-3: Memory exhaustion** (fragment.c:137)
- Unbounded reassembly entries
- Attacker can flood with unique IDs
- Fix: Limit to 16 max entries
- **STATUS: FIXED**

**NEW-1: Bitmask calculation order** (fragment.c)
- `(1 << total_frags)` calculated before checking total_frags <= 8
- Risk: VERY LOW (earlier validation prevents issue)
- Fix: Move validation before calculation
- **STATUS: FIXED (non-blocking)**

**Build Testing Critical:**
- Code compiled on Mac ≠ code compiles on Pi
- Platform-specific headers (sys/types.h for ssize_t)
- Compiler strictness varies (arm64 gcc more strict)
- Always test on target platform before declaring success

**GPIO Management:**
- GPIO pins can be held by previous processes or device-tree overlays
- Use `gpioinfo` to check pin status
- Use `gpioset --mode=exit` to release pins
- libgpiod cleaner than deprecated /sys/class/gpio
