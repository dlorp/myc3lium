ISSUES FIXED
------------
✓ CRITICAL-1: Buffer overflow in fragment reassembly (fragment.c:187)
✓ CRITICAL-2: Missing fragment index validation (fragment.c:109)
✓ CRITICAL-3: Signal handler deadlock risk (lora-tap-bridge.c:38)
✓ HIGH-3: Payload length overflow in fragment_encode (fragment.c:81)
✓ HIGH-4: SPI buffer overflow in sx1262_write_command (sx1262.c:195)
✓ HIGH-5: Off-by-one buffer overflow in sx1262_transmit (sx1262.c:468)
✓ MEDIUM-3: Memory exhaustion DoS (fragment.c:137)

CHANGES SUMMARY
---------------

CRITICAL-1 (fragment.c):
  - Added validation: total_frags must be <= MAX_FRAGMENTS
  - Added bounds check: frag_index must be < MAX_FRAGMENTS
  - Prevents buffer overflow via malicious fragment packets

CRITICAL-2 (fragment.c):
  - Added fragment index validation in fragment_decode()
  - Rejects packets with frag_index >= MAX_FRAGMENTS at decode time
  - Fail-fast approach prevents downstream buffer issues

CRITICAL-3 (lora-tap-bridge.c):
  - Changed 'running' flag type to sig_atomic_t
  - Removed printf() from signal handler (async-unsafe)
  - Moved shutdown message to main loop after exit
  - Now fully POSIX async-signal-safe

HIGH-3 (fragment.c):
  - Added payload_len validation in fragment_encode()
  - Rejects payloads > FRAG_MAX_PAYLOAD (252 bytes)
  - Protects against corrupted fragment structures

HIGH-4 (sx1262.c):
  - Added size validation in sx1262_write_command()
  - Rejects commands with len > 255
  - Prevents stack buffer overflow in SPI transfers

HIGH-5 (sx1262.h):
  - Reduced SX1262_MAX_PAYLOAD from 255 to 254
  - Prevents off-by-one overflow in sx1262_transmit()
  - 2-byte header + 254 bytes payload = 256 bytes (fits in buffer)

MEDIUM-3 (fragment.h, fragment.c):
  - Added MAX_REASSEMBLY_ENTRIES = 16 constant
  - Modified find_or_create_entry() to enforce limit
  - Prevents memory exhaustion DoS attacks
  - Maximum memory usage: 16 × 1508 bytes ≈ 24 KB
